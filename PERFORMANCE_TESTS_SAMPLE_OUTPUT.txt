1. For batch processing (no UI): Use Headless mode
   - Lowest overhead, best throughput
   - Recommended batch size: 10,000-50,000

2. For read-only grids: Use Readonly mode
   - UI dispatcher available but inactive
   - Recommended batch size: 5,000-10,000

3. For interactive editing: Use Interactive mode
   - Full UI refresh with auto-update
   - Recommended batch size: 1,000-5,000

4. For 1M+ rows:
   - Prefer Headless or Readonly modes
   - Use larger batch sizes (10,000+)
   - Enable parallel processing
   - Disable real-time validation

5. Memory optimization:
   - Use streaming for 10M+ rows
   - Disable caching if memory-constrained
   - Process data in chunks

KEY INSIGHTS FROM THIS SAMPLE:

1. BATCH SIZE SWEET SPOT:
   - Headless: 10K-50K (best throughput)
   - Readonly: 5K-10K (balanced)
   - Interactive: 5K (best UX)

2. MODE OVERHEAD:
   - Readonly: ~10% slower than Headless
   - Interactive: ~45% slower than Headless

3. OPERATION PERFORMANCE (100K rows):
   - GetAllRows: Fastest (~40ms) - simple retrieval
   - UpdateCells: Very fast (~130ms) - 1% updates
   - Filter: Fast (~650ms) - efficient filtering
   - BulkInsert: Fast (~800ms) - raw insertion
   - Sort: Medium (~1,200ms) - comparison overhead
   - Validation: Slowest (~2,400ms) - regex processing

4. SCALING TO 1M ROWS:
   - Sort: ~10-13x slower (expected for O(n log n))
   - Filter: ~12-15x slower (linear scan)
   - Validation: ~15-18x slower (regex overhead)
   - BulkInsert: ~10-12x slower (memory allocation)

5. MEMORY USAGE:
   - Scales roughly linearly with row count
   - Interactive mode uses ~20% more memory
   - Validation uses most memory (regex state)
